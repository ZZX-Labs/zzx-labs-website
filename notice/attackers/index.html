Attackers, Abuse & Security Incident Register

This page is reserved for documenting known security attacks, hostile automation, malicious traffic, blocked abuse sources, banned IP ranges, attempted intrusions, spam campaigns, scanning behavior, credential abuse, scraping abuse, denial-of-service patterns, and other activity that targets ZZX-Labs R&D infrastructure.

Entries should be evidence-based, minimal, defensive, and operational. This page is not for harassment, retaliation, public shaming, speculation, doxxing, or unverified accusation.

Security Notice

Defensive Documentation Only

This register exists to support security operations. Any record added here should help ZZX-Labs identify, block, investigate, or understand hostile activity. Records should be factual and conservative.

Do not publish private personal information, unverified claims, revenge statements, threats, leaked credentials, exploit instructions, or information that would help attackers reproduce an incident.

Incident Categories

What May Be Tracked

Malicious Scanning

Port scans, path probing, admin-path discovery, vulnerability scanning, and suspicious enumeration.

Brute Force

Repeated login attempts, credential stuffing, password spraying, SSH abuse, or API authentication abuse.

Spam / Form Abuse

Contact spam, fake submissions, bot comments, malicious links, phishing attempts, or garbage payloads.

Scraping Abuse

Excessive scraping, hostile mirroring, bandwidth abuse, bot traffic, and policy-violating automation.

Denial-of-Service

Traffic floods, abusive request rates, repeated large-file hits, resource exhaustion, or coordinated disruption.

Confirmed Intrusion

Any verified unauthorized access, file modification, credential exposure, malware, or infrastructure compromise.

Public vs Private

What Should Be Published

Public

High-level incident category, date range, general impact, defensive action taken, and sanitized lessons learned.

Private

Raw logs, credentials, full IP lists, internal paths, server configuration, detection rules, and exploit details.

Redacted

IP addresses, user agents, request payloads, hostnames, identifiers, and sensitive evidence unless disclosure is justified.

Escalated

Serious attacks may be escalated to hosting providers, registrars, CERTs, law enforcement, counsel, or trusted partners.

Banned IP Policy

Blocking and Banning

IP bans should be treated as operational controls, not final attribution. A single IP address may represent a VPN, proxy, NAT gateway, cloud provider, compromised device, scanner, shared network, or unrelated user.

Public ban lists should be used carefully. Internal blocklists may be more detailed than public records. Public entries should avoid implying identity unless attribution is verified through reliable evidence.

Severity Model

Security Severity Levels

Critical

Confirmed compromise, credential exposure, private data leak, payment-system risk, or active intrusion.

High

Exploit attempt against sensitive service, repeated authentication attacks, major abuse, or likely targeted activity.

Medium

Scanning, probing, repeated suspicious behavior, aggressive bots, spam abuse, or policy-violating scraping.

Low

Noise-level internet scanning, malformed requests, low-volume spam, or isolated suspicious traffic.
