Bug Bounty Program

This page defines the future ZZX-Labs R&D bug bounty and responsible security testing program. It exists to guide researchers, contributors, testers, and security reviewers who discover vulnerabilities, broken workflows, exposed data, unsafe configurations, or weaknesses in public ZZX-Labs systems.

The program is currently a policy placeholder. Rewards, scope, payout structure, legal safe-harbor language, and eligible systems must be finalized before any paid bounty claim is considered active.

Program Status

Policy Placeholder

ZZX-Labs does not yet operate a fully active paid bug bounty program. This page establishes the intended structure so security researchers know how ZZX-Labs expects vulnerabilities to be handled.

Until a formal bounty schedule is published, reports may be accepted as responsible disclosures but should not be assumed eligible for payment, reward, credit, or public recognition.

Responsible Testing

Rules of Engagement

  • Do not access, alter, delete, or exfiltrate private data.
  • Do not attempt persistence, lateral movement, malware deployment, phishing, or social engineering.
  • Do not perform denial-of-service testing, stress testing, flooding, or resource exhaustion.
  • Do not test third-party systems unless they are explicitly listed in scope.
  • Do not publish vulnerability details before ZZX-Labs has reviewed and resolved the issue.
  • Use the minimum proof necessary to demonstrate impact.
  • Stop testing immediately if private data, credentials, keys, or sensitive systems are encountered.
  • Report findings through the official contact path.

Potential Scope

Future In-Scope Areas

Public Website

Static pages, navigation, asset paths, client-side scripts, public JSON, and public documentation.

Public Tools

Bitcoin widgets, Bitnodes pages, mempool pages, CyberChef integrations, dashboards, and public utilities.

Public APIs

Only APIs explicitly listed as public and in scope after formal activation.

Release Workflows

Checksums, signing keys, release manifests, download paths, and public verification instructions.

Generated Pages

Startup, staff, partner, consultant, notice, project, and documentation generation workflows.

Security Notices

Canary, breach, alert, disclosure, and key-history pages.

Out of Scope

Not Authorized

  • Physical attacks, theft, trespass, coercion, or harassment.
  • Social engineering against staff, partners, contributors, vendors, or users.
  • Phishing, credential theft, malware, persistence, or command-and-control activity.
  • Denial-of-service, load testing, spam, brute force, scraping abuse, or destructive testing.
  • Testing systems owned by third parties without permission.
  • Accessing private infrastructure, private email, storage, admin portals, or unpublished services unless explicitly authorized.
  • Public disclosure before coordinated review.

Severity

Preliminary Severity Categories

Critical

Private key compromise, payment address substitution, authentication bypass, private data exposure, or release-signing compromise.

High

Unauthorized access, sensitive data leak, serious API flaw, stored XSS in trusted context, or serious workflow compromise.

Medium

Reflected XSS, path issues, data validation gaps, weak headers, broken access assumptions, or noncritical data exposure.

Low

Information leakage, minor misconfiguration, low-risk client issue, typo affecting security guidance, or weak documentation.

Future Rewards

Bitcoin-Native Reward Model

If a paid bounty program is activated, ZZX-Labs may publish Bitcoin-denominated reward ranges based on severity, exploitability, report quality, scope, novelty, and actual impact.

Any future reward model should be explicit, published in advance, and tied to confirmed eligible findings.
