Purpose

What This Page Is For

This canary page is designed to communicate high-level transparency information without exposing sensitive operational details. It may eventually state whether ZZX-Labs has received certain categories of legal process, gagged orders, secret demands, infrastructure compromise notices, or other exceptional conditions to the extent legally permitted.

This page should not be used to publish private legal strategy, internal investigations, raw logs, security secrets, user data, credentials, or unverified claims.

Canary Language

Unsigned Draft Statement Template

The following block is a template. Do not treat it as active unless it has a real date, active signing key, detached signature, and matching hash. 

ZZX-Labs R&D Warrant Canary Statement

Date: YYYY-MM-DD
Valid Until: YYYY-MM-DD
Statement ID: CANARY-YYYY-MM

As of the date above, to the extent legally permitted to say so:

1. ZZX-Labs R&D has not received a secret court order requiring undisclosed surveillance.
2. ZZX-Labs R&D has not received a national-security-style gag order that prevents disclosure.
3. ZZX-Labs R&D has not been compelled to add hidden access, backdoors, or secret monitoring to public releases.
4. ZZX-Labs R&D is not aware of an undisclosed breach affecting private user data outside public open-source repositories.
5. ZZX-Labs R&D has not knowingly published compromised release artifacts.
6. ZZX-Labs R&D has not knowingly lost control of its active signing key.

This statement should be verified against the current active public signing key listed in contact/key-history/.

Fingerprint: [ACTIVE_KEY_FINGERPRINT]
SHA256: [STATEMENT_SHA256]
Signature: [DETACHED_SIGNATURE_PATH]

Verification

How Canary Verification Should Work

  1. 01Publish Statement: Place the canary text in a stable file path.
  2. 02Hash Statement: Publish SHA256 and SHA3-256 hashes of the exact statement file.
  3. 03Sign Statement: Sign with the active ZZX-Labs release/signing GPG key.
  4. 04Archive Version: Store old canaries permanently in a chronological archive.
  5. 05Update Key History: If keys rotate, record the change in contact/key-history/.
  6. 06Expire Clearly: Every canary should have a clear valid-until date.

Interpretation

How To Interpret A Missing Canary

A missing, expired, delayed, altered, unsigned, or unverifiable canary should not automatically be interpreted as proof of compromise or legal coercion. It may also result from operational delay, migration, key rotation, infrastructure failure, or administrative error.

However, consistent failure to update a canary should be treated as a signal requiring review. Users should compare the current statement, prior archives, signatures, fingerprints, hashes, release notes, key-history entries, and breach notices.
