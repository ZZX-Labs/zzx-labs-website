Bitcoin Node Latency

This page tracks latency and timing behavior across the ZZX-Labs Bitnodes mirror. Latency records help estimate how quickly reachable Bitcoin nodes respond during crawler checks, how regional networks behave, how peer responsiveness changes over time, and how efficiently the crawler is gathering public node data.

Latency should be interpreted as crawler-observed response timing, not as a universal measure of the Bitcoin network. Results depend on crawler location, network path, timeout settings, public node behavior, Tor/I2P routing, DNS seed quality, peer availability, and the specific snapshot window.

Latency Bands

Response Timing Buckets

< 100 ms

Very fast response timing. Often local, regional, well-peered, or hosted on strong infrastructure near the crawler path.

100–250 ms

Healthy response timing for many public internet paths, especially across regions or cloud networks.

250–500 ms

Moderate latency. May reflect distance, routing, node load, home internet, Tor paths, or temporary network congestion.

500–1000 ms

Slow but still responsive. These nodes may be reachable yet degraded, distant, filtered, overloaded, or routed through poor paths.

> 1000 ms

High latency. Treat as degraded unless explained by Tor, satellite, mobile, experimental routing, or crawler-side network limits.

Timeout

No usable response before the crawler timeout. This does not always mean the node is permanently offline.

Crawler Timing

What Latency Means Here

The Bitnodes crawler measures timing from the crawler’s perspective. This is useful for trend monitoring, identifying unusually slow peers, comparing regions, and improving crawler efficiency, but it is not a universal truth about every user’s network path to a node.

Connect Timing

Time needed to establish a connection to a public peer, subject to timeout settings and route quality.

Handshake Timing

Time required to complete enough Bitcoin protocol interaction to classify the peer as responsive.

Round-Trip Behavior

Observed response behavior during crawler checks, influenced by distance, congestion, routing, node load, and filtering.

Timeout Rate

Percentage of candidate nodes that fail to respond within the crawler window or configured timeout.

Retry Behavior

Repeated attempts may turn failed candidates into reachable nodes, especially during temporary congestion or peer churn.

Window Bias

Latency can vary by crawl window, time of day, crawler runner location, DNS seed results, and network conditions.

Optimization

Why Latency Helps The Crawler

Better Timeouts

Latency history helps tune timeout values without losing too many slow but reachable peers.

Smarter Rechecks

Slow or unstable nodes can be rechecked less aggressively than consistently responsive peers.

Regional Awareness

Regional latency differences help distinguish real node problems from distance or route effects.

Cleaner Public API

Aggregated latency files keep the website fast while raw crawler snapshots stay out of normal page loads.

Peer Quality

Latency can become one component of peer health alongside uptime, recency, services, version, and reachability.

Research Value

Longitudinal timing data can support reports on Bitcoin network geography, hosting concentration, and public node resilience.
