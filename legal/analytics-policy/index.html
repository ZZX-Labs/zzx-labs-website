Analytics Policy

This Analytics Policy explains how ZZX-Labs R&D may measure website usage, page availability, tool performance, API behavior, download activity, widget usage, errors, traffic patterns, and security events.

ZZX-Labs R&D aims to use analytics for operational visibility, security, debugging, performance, research, and infrastructure planning, not for invasive advertising or sale of user behavior.

1

Purpose of Analytics

Analytics may be used to understand whether the website works, which pages are broken, which public tools are useful, which downloads are active, which widgets fail, which APIs need improvement, and whether traffic patterns suggest abuse or security risk.

Analytics should support maintenance, security, quality assurance, capacity planning, debugging, and public infrastructure improvement.

2

Information That May Be Measured

  • Page views
  • Requested URLs
  • Referring pages
  • Browser type
  • Operating system type
  • Device class
  • Screen-size category
  • Approximate request time
  • HTTP status codes
  • Page load errors
  • JavaScript errors
  • Broken asset requests
  • API endpoint usage
  • Widget load success or failure
  • Download counts
  • Security and abuse events

3

Information ZZX-Labs Does Not Intend to Sell

ZZX-Labs R&D does not intend to sell personal analytics data, behavioral profiles, visitor records, contact submissions, security reports, bug reports, donor communications, or private service inquiries to advertisers or data brokers.

4

Cookies, Local Storage, and Client-Side State

Cookies, local storage, session storage, or similar browser-side mechanisms may be used for site functionality, interface preferences, debug state, dashboard configuration, widget settings, or future account systems.

Where possible, analytics should minimize persistent identifiers and avoid unnecessary tracking.

5

Server Logs

Web servers, API servers, reverse proxies, static hosts, and future infrastructure may collect logs as part of normal operation. Logs may include IP addresses, timestamps, requested paths, status codes, user agents, referrers, and error messages.

Logs may be used for debugging, uptime analysis, security monitoring, abuse prevention, rate limiting, and infrastructure planning.

6

Security Analytics

Security-related analytics may be collected to detect malicious scanning, credential attacks, phishing, spam, scraping abuse, denial-of-service attempts, suspicious automation, exploit attempts, fraudulent payment behavior, and other hostile activity.

Security logs may be retained longer than ordinary analytics where necessary to protect infrastructure, investigate abuse, preserve evidence, or comply with legal obligations.

7

Bitcoin Widget and API Metrics

Bitcoin tools, BPI widgets, Bitnodes pages, mempool pages, mining widgets, payment tools, exchange-rate tools, calculators, charts, and future APIs may collect operational metrics to determine uptime, data-source reliability, error rates, usage levels, and performance.

These metrics should be used to improve tools and infrastructure rather than to build invasive visitor profiles.

8

Download and Release Metrics

ZZX-Labs R&D may measure downloads of software, papers, datasets, documentation, media, models, archives, reports, and public releases.

Download metrics may help prioritize maintenance, determine user interest, identify broken release paths, and evaluate infrastructure capacity.

9

Third-Party Analytics and Hosting

Public pages, repositories, downloads, or infrastructure may be hosted through third-party services such as static hosting providers, repository platforms, DNS providers, CDN providers, security services, or API providers.

Third-party services may collect their own logs or analytics under their own policies. ZZX-Labs R&D does not control every third-party analytics process.

10

Retention

Analytics data may be retained for operational, debugging, research, historical, security, legal, or infrastructure purposes. Retention periods may vary depending on data category and risk level.

Where practical, analytics data should be aggregated, anonymized, minimized, or deleted when no longer needed.

11

Opt-Out and Browser Controls

Users may control some tracking technologies through browser settings, extensions, privacy tools, cookie controls, script blocking, network filtering, or privacy-focused browsers.

Disabling scripts, cookies, or local storage may affect some site features, widgets, dashboards, forms, and future account functions.

12

Future Account and API Analytics

Future account systems, API tolls, private services, secure communications, research portals, and Bitcoin payment systems may require additional analytics for authentication, billing, abuse prevention, reliability, rate limiting, fraud detection, and audit trails.

Additional policies or service-specific terms may apply once those systems are built.

13

Policy Updates

This Analytics Policy may be updated as analytics systems, tools, widgets, APIs, private services, and public infrastructure evolve. Updated versions should be versioned and archived through the Legal Hub.
