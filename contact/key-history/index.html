GPG Key History

This page records the public signing-key history for ZZX-Labs R&D, 0xdeadbeef, and related laboratory release infrastructure. It exists so users, collaborators, researchers, contributors, and future clients can verify which cryptographic identities are active, transitional, expired, discontinued, or no longer trusted for new releases.

These keys are intended to cover signatures for public website releases, software, firmware, hardware documentation, applications, AI and ML models, databases, datasets, papers, reports, downloads, release manifests, hashes, checksums, detached signatures, and other published ZZX-Labs artifacts.

Green Active

Active keys are current standard signing keys. These are the preferred keys for verifying present ZZX-Labs releases and communications.

Yellow Transitional

Transitional keys may be used during migration, overlap, staged rotation, emergency replacement, or multi-system upgrade periods. A transitional key may later become active.

Red Retired

Retired keys are expired, discontinued, replaced, revoked, or no longer used for new releases. They may remain listed for historical verification of older signatures.

Policy

Why This Page Exists

ZZX-Labs expects signing keys to change periodically as part of security hardening. Rotation creates a public record of operational discipline: when keys were changed, why they were changed, which keys signed which artifacts, and when users should stop trusting older keys for new releases.

Frequent key rotation makes the release chain more resilient, but only if the history is visible. This page therefore acts as a public ledger for signing identity changes across the website, tools, software, firmware, hardware documentation, AI and ML artifacts, databases, datasets, and all other published materials.

Verification Rule

Do Not Trust Unlisted Release Keys

Any ZZX-Labs release-signing key should be listed here before it is treated as authoritative. If a signature, checksum, release manifest, model card, firmware package, application installer, dataset, or paper claims to be signed by ZZX-Labs but the key is not listed on this page, treat that artifact as unverified until it is confirmed through the current active key or another authenticated channel.

Rotation Standard

Recommended Key Lifecycle

ZZX-Labs should treat key rotation as a planned operational event, not an emergency-only action. A new key may begin as yellow transitional, overlap with the current green active key, and later become green once it is fully published, tested, backed up, verified, and used across release workflows. The prior key then moves to red retired status unless it remains intentionally transitional.

Emergency rotation should occur immediately if a private key is suspected to be exposed, lost, copied, mishandled, stored insecurely, signed from an untrusted machine, or used in a compromised release process.
