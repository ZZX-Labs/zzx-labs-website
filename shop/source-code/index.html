Release Metadata

What Each Code Release Should Include

README

Purpose, features, installation, usage, dependencies, warnings, and project status.

LICENSE

Clear license file and reuse rights for the codebase.

CHANGELOG

Versioned changes, fixes, breaking changes, and migration notes.

Requirements

Dependency files such as requirements.txt, package.json, go.mod, or build configs.

Tests

Validation scripts, unit tests, integration tests, fixtures, and sample data.

Docs

Manuals, API references, examples, diagrams, screenshots, and architecture notes.

Hashes

Checksums for release archives, binaries, installers, source bundles, and datasets.

Signatures

Signed tags, detached signatures, GPG verification notes, and key-history references.

Licensing

Source Code Licensing

Source code may be released under different licenses depending on project goals, dependencies, upstream licenses, commercial needs, security constraints, and legal review.

MIT

Preferred permissive default for many original ZZX-Labs software releases.

BSD / Apache

Permissive licenses suitable for libraries, tools, and reusable infrastructure.

GPL-Compatible

Used where upstream dependencies, community norms, or copyleft goals require it.

Commercial

Paid commercial-use rights, enterprise licensing, support contracts, and private distributions.

Research Use

Limited-use code packages for education, research, review, and controlled testing.

Restricted

Security-sensitive or client-specific code may require manual approval and written terms.

Security

Source Code Security

Source releases should never include private keys, seed phrases, credentials, tokens, passwords, API secrets, sensitive infrastructure paths, private customer data, internal-only documents, or exploit-enabling material that has not been reviewed for safe publication.

Secret Scanning

Check for keys, tokens, passwords, private config, wallet data, and credentials before release.

Dependency Review

Review third-party packages, license compatibility, supply-chain risk, and pinned versions.

Safe Defaults

Avoid insecure default credentials, exposed admin interfaces, and unsafe production settings.

Responsible Disclosure

Security issues should be reported privately through official disclosure channels.
